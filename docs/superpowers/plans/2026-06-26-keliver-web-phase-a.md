# keliver Web — Phase A: Make the Web Target First-Class

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Replace the `spike/keliver-web` build hacks with a clean, opt-in `WasmJs` target modifier so the web (Compose-for-Web / Wasm) is a first-class, non-hacky target that CI protects.

**Architecture:** Three moves: (1) a `WasmJs` opt-in `TargetModifier` (mirroring the existing `JsTests`) applied per render-chain module — deleting the `WASM_EXCLUDED` denylist and the unconditional `wasmJs { browser() }` from shared group branches; (2) consolidate the duplicated `wasmJs` actuals into shared `webMain` (js + wasmJs) source sets; (3) add `apiDump` baselines for the new Wasm klib variants and a `compileKotlinWasmJs` CI check so web never silently breaks.

**Tech Stack:** Kotlin Multiplatform, Gradle, Compose Multiplatform 1.8.2 (publishes `-js` + `-wasm-js`), binary-compatibility-validator (klib), GitHub Actions.

**Branch:** `spike/keliver-web` (NOT main). All builds need `JAVA_HOME=$(/usr/libexec/java_home -v 17)`.

**Verification note:** This is build-config work; the "tests" are gradle compile/apiCheck tasks, not unit tests. Each task ends with a real gradle gate whose output must be observed (per superpowers:verification-before-completion) before checking the box.

- [ ] **Step 1: Add the `WasmJs` modifier** in `RedwoodBuildExtension.kt`, directly after the `JsTests` enum (around line 110). It mirrors `JsTests` but targets the multiplatform extension (wasmJs is an *added* target, not a sub-environment of an existing one):

```kotlin
enum class WasmJs : TargetModifier {
  Browser,
  ;

  fun applyTo(kotlin: org.jetbrains.kotlin.gradle.dsl.KotlinMultiplatformExtension) {
    when (this) {
      Browser -> kotlin.wasmJs { it.browser() }
    }
  }

  override val key get() = Companion
  companion object : TargetModifier.Key<WasmJs>
}
```

## Task 2: Consolidate the duplicated wasmJs actuals into `webMain`

The spike put platform actuals only in `wasmJsMain`. Where a parallel `jsMain` actual already exists (the module had `js()` before the spike) and the two are byte-identical, hoist into a shared `webMain` (parent of `jsMain` + `wasmJsMain`) so there is one copy.

**Files:** `keliver-runtime`, `keliver-composeui`, `keliver-leak-detector` (the three modules with `wasmJsMain` actuals).

- [ ] **Step 1: Inventory js-vs-wasm actual pairs.**

Run: `cd /Users/sanchitwalia/AndroidStudioProjects/konduit && for m in keliver-runtime keliver-composeui keliver-leak-detector; do echo "== $m =="; find $m/src/jsMain $m/src/wasmJsMain -name '*.kt' 2>/dev/null; done`
For each `wasmJsMain` file, note whether a same-named `jsMain` actual exists and whether `diff` shows them identical.

- [ ] **Step 2: For each module that has BOTH js + wasmJs actuals, define a `webMain` group** in that module's `build.gradle` (follow `keliver-compose`'s custom-hierarchy pattern at `keliver-compose/build.gradle:16-60`). Add to the `applyDefaultHierarchyTemplate`/custom hierarchy a `web` group containing `js` + `wasmJs`, and create `src/webMain/kotlin/...`. Move the identical actual there; delete the `jsMain` + `wasmJsMain` copies.

(Only do this where the js and wasm actuals are identical. If they diverge — e.g. js uses array-interop and wasm uses plain-Kotlin — leave them split; that divergence is the whole reason `keliver-compose` has its `nonJs` group, and forcing a merge would be wrong.)

Run: `JAVA_HOME=$(/usr/libexec/java_home -v 17) ./gradlew :keliver-runtime:compileKotlinJs :keliver-runtime:compileKotlinWasmJs :keliver-composeui:compileKotlinJs :keliver-composeui:compileKotlinWasmJs :keliver-leak-detector:compileKotlinJs :keliver-leak-detector:compileKotlinWasmJs --console=plain`
